JavaWeb 使用第三方 Jar 的问题。

这个是我自己亲自体验到的问题，想着这里记录一下，在普通项目我们导入 Jar包会将其加入到当前项目的 ClassPath 中，但是 JavaWeb 项目呢？也是添加 ClassPath 吗？

之前我找了一下网上的答案，貌似并没有提 ClassPath 这个概念，而是提出了一种曲线救国的方式，将第三方 Jar 包添加到 Tomcat 的 lib 目录下，这样也行是没问题的，因为 lib 文件内的 jar 包肯定是在 Tomcat 自己的类加载路径下的嘛。

但是这样有一个问题，就是所有的项目默认其实都能够访问 所增加的 jar 包了，虽然类是动态加载，不会主动的加载 jar 包，但是总感觉不是最完美的方式。就好比普通的 java 项目在导入第三方 jar 包的时候，也没见往 java 的内置包路径中添加啊。

所以我找了一下针对于当前 Tomcat 容器的 ClassPath，应该也存在着这样一个类加载的路径，在这里设置好你需要 Jar 的时候，这样你在发布的时候 Tomcat 就知道去哪找你引用的第三方 jar 包了。当然这只是让你能够在本地运行，如果你放到其他地方运行，应该跟 Tomcat 的设置有关，挖个坑吧，后面有时间了解一下 Tomcat。

问题二

/WEB-INF、/target、/META-INF、/Server 这几个目录到底是干啥的。

首先 /WEB-INF 和 /META-INF 这两个目录我们在上面已经说过啦，这里就聊聊 /target 和 /Server 目录啦。

/target

/target 目录其作用就是存放编译后的 class 文件啦，具体我就不截图啦，大家可以自己试试。现在最重要的是为什么会放到这里呢？

首先 class 文件是通过 javac 命令生成的，javac 可以选择带上输出路径的参数，如果不加这个参数会默认输出到源文件目录，如果带上就会指定到指定目录。所以现在为什么 class 会出现在 /target 目录下我们就知道了，一定是有一个地方指定了输出路径，现有又有另一个问题，在哪个地址指定的呢？

我们先说说一个图形化的地方，其实就是放到 BuildPath 的 Source 标签中啦：

当然这些显式内容都存在一个配置文件，这个输出目录其实就是对应项目的 `.classpath 文件

同理你会发现好多项目会把编译后的文件放到 /build 目录中，同样的道理咯。
