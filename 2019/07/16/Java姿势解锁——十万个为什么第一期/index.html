看完了 public 修饰类的效果之后我们回过头来看这个问题，为什么呢？这里我上网上找到了一个答案：

声明为public的类可以被包外的类引用，只需要import 包名，就可以使用该类。这里我要指出：编译器在执行这条语句时会根据路径反向查找，而此时的文件名能帮助编译器快速定位所编译类的位置。

以下开始 HL 的瞎BB

我们回想一下之前所讲的类加载过程： Java姿势解锁 —— 类加载过程，它是动态进行加载的。那么编译呢？是不是也存在着动态编译？

再次说一下，以下都是HL的瞎BB。

当我们在一个类的开头部分定义 import 包名 这样的声明，在使用到对应的类的时，如果其 java 文件还未进行编译的话，这时候需要先对其源代码文件进行编译，而 javac 命令是需要知道 .java 文件的路径的，那么如何找到当前我们需要类的源文件呢？

比如我们一个程序需要用到 com.lalala.biubiubiu 包下的 WakakaClass 类，如果此时 类名和文件名一致，这时候就可以直接得出源文件的路径为 com/lalala/biubiubiu/WakakaClass.java，如果不一致，可能需要把当前包下所有的文件都编译一下，然后找到编译后 的 Class 文件为 WakakaClass.class。

因为存在被第三方用到的可能，我们需要将类使用 public 修饰。为了使编译期更加快速定位到源文件的位置，我们需要保持其类名和文件名一致。

附加问题

一个 java 文件中能否有多个 public 修饰的类？

这个问题首先可以用上一个问题的结论来解决：一个源文件中，使用 public 修饰的类名称必须与文件名相同，那么文件名只有一个，同一包下的类名也不允许重复，那么自然一个 java 文件中也只能拥有一个 public 修饰的类咯，这个类名还必须与文件名相同。

结论是这样，不过，为什么呢？以下结论网上大佬们的回答：

每个编译单元(文件)只能有一个public类。这么做的意思是，每个编译单元只能有一个公开的接口，而这个接口就由其public类来表示。而非public修饰的类都是为了给public修饰的类所做支撑的，从软件架构设计和安全性设计上得出的结论。或者说是java的设计者们从这方面的考虑。或许这真的是一个规范。

问题二

为什么 static 修饰的方法 不能访问 非static 修改的实例变量呢？

举个栗子：

TestClass 类的静态方法 getName 想访问非静态变量 name，这时候编译会报错。这个问题其实配合着我们之前聊到的类的存储相关知识就能理解啦。
首先静态资源(包括方法和静态变量)是随着类一起存在的，也就是在类加载的时候就会存在于方法区中，并且也会创建对应的 Class 对象存放于堆中，其中保存着方法的一些信息。但是实例对象是跟对象一起存储于堆中的，也就是说如果没有实例化类的话，成员变量就是不存在的，注意这里是没有分配内存，简单说就是没有办法被访问的。
再回到上面那个例子，调用 TestClass 类的静态方法 getName() 时候，这个时候 TestClass 类根本没有实例化呀，所以这个静态方法访问这一块根本没有存在的内存地址是一定会报错的嘛，Java 编译器知道这一点于是就报错咯。

问题三

经常看到好多方法的参数 或是 方法本身被 final 修饰，这是为什么呢，这样做的好处有什么？

以下的部分内容来源于其它博主的博客，全部都可以在 Ref 中找到。

final 修饰类

被 final 修饰的类无法被修改，无法被继承，使用 final 修饰类主要出于以下目的：

  1. 这个类已经是一个十分完善的类，我们不会对其进行任何的改变，不希望这个类被任何人继承改变它。
  2. 出于安全考虑，我们不希望进行子类化处理。
  3. 我们希望jvm在处理的时候，不再关心类是否被继承，jvm不需要更多额外的处理，提高执行效率。
  4. 将类定义成 final后，结果只是禁止进行继承——没有更多的限制。然而，由于它禁止了继承，所以一个 final类中的所有方法都默认为 final。final方法在编译阶段绑定，称为静态绑定(static binding),编译时常量，永远不可改变。jvm在运行的时候，会直接将final方法直接嵌进来，效率会大大提升。

关于这里的第四点我们将在后面 final 修饰方法部分聊一聊。

final类的对象

  • 不可变类指它的对象一旦被创建了就不能被更改了
  • final类的对象是只读的
  • 可以在多线程环境下安全的共享
  • 不用额外的同步开销

final 修饰方法

同样是 final 关键字的策略，被 final 修饰的方法是不允许被修改的，当然在继承关系中被 final 修饰的方法是不允许被重写的。使用 final 修饰方法主要出于以下目的：

  1. 一个方法的功能已经足够完整了，子类中不需要改变的话，就声明此方法为final。 它会为方法“上锁”，防止任何继承类改变它的本来含义。设计程序时，若希望一个方法的行为在继承期间保持不变，而且不可被覆盖或改写，就可以采取这种做法。
  2. final方法比非final方法要快，因为在编译的时候已经静态绑定了，不需要在运行时再动态绑定。将一个方法设成 final 后，编译器就可以把对那个方法的所有调用都置入“嵌入”调用里。只要编译器发现一个final 方法调用，就会（根据它自己的判断）忽略为执行方法调用机制而采取的常规代码插入方法（将自变量压入堆栈；跳至方法代码并执行它；跳回来；清除堆栈自变量；最后对返回值进行处理）。相反，它会用方法主体内实际代码的一个副本来替换方法调用。这样做可避免方法调用时的系统开销。

这里我们主要说一下第二点，首先关于什么是静态绑定和动态绑定这里我就不解释了，可以参考一下这边博客：Java静态绑定与动态绑定

那么静态绑定的方法执行性能真的比动态绑定的方法好吗？简单说就是 final 修饰的方法执行性能要比非 final 修饰的的方法好？带着疑问我看到了这篇文章：Java中final方法比非final性能更好?

有兴趣的童鞋可以去看看，这里我就说一下博主的结论吧：使用 final 的确是有提升的，但是几乎可以忽略不计，特别是放到现在高性能的 JVM 中，它们生成的字节码除了 flags 标志位是否有 final 修饰不同之外，其他所有的JVM指令，都是一样的（对于方法本身，以及调用者本身的字节码都一样）。对于JVM来说，它执行的就是字节码，如果字节码都一样的话，那对于JVM来说，它就是同一样东西的了。

fianl 修饰变量

被 final 修饰的变量是不可变的，这一点不多说啦，使用 final 修饰变量最只要的目的是：

多线程情况下的数据共享安全！

大家可以看看知乎一位大佬的回答：JVM对于声明为final的局部变量（local var）做了哪些性能优化？，简单总结一下：

final 修饰变量主要用途是保证多线程情况下的数据共享安全，比如：在同一个方法里连续两次访问静态变量A.x可能会得到不一样的值，因为可能会有并发读写；但如果先有final int x = A.x然后连续两次访问局部变量x的话，那读到的值肯定会是一样的，但是对于性能是没有什么提升的。(大佬们都是研究编译后的字节码的吗…..)
