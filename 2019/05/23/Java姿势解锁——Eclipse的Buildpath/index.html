发表于 2019-05-23 • 分类于 流下了没有技术的泪水 , Java姿势解锁

前言

这次说说 Eclipse 的 Buildpath。

这东西在第一次使用 Eclipse 的时候就产生了疑惑，这里终于下定决心去了解一下，看看内部到底是怎么运作的。目前的打算是将这部分做成一个小系列吧，主要就是去弄懂身边常常被忽视的小知识，应该还会有后期的(小声BB)

Buildpath

第一次在 Eclipse 中需要引入第三方 Jar 包的时候，回想一下我们的操作：

  1. 将 Jar 复制到 Lib 目录下
  2. 右击选择 Add to Build Path

那么这个 BuildPath 到底是个什么东西啊？为什么需要加到 BuildPath才能使用呢？

要回答上面的问题我们可以换一种方式来思考：Eclipse 总归是一个 IDE，它的所有操作其实追其本质都是对 Java 命令的操作，只是它将 Java 的命令图形化了，当我们点击上面按钮，添加配置，它就会去地调用Java 相应的命令完成操作。

那么可以这样思考问题，当我们抛开IDE的时候，如何引用第三方jar包进行程序的编写呢？

这里 -cp 指定了 jar包所在目录，但是后面接了一个 : 号，这是什么意思呢。这里的冒号表示分隔符，你可以通过它指定多个目录。比如你引入了 5个jar包，它们分别位于5个不同的目录，这时候你就需要指定5个jar包的位置啦。

通过这个栗子我们做出如下理解
Eclipse 的 Buildpath 其实就是保存第三方 jar 包的位置，这样当我们需要使用 第三方 jar 包的时候，就将其添加到 Buildpath 中，Eclipse 就会自动去这个 Buildpath 中寻找，如果没有找到的话当然就报错啦。

注意这里 Eclipse 会有编辑的检测，所以其实我们在编译的时候就需要将 jar 包加入 Buildpath 中，这样 Eclipse 就知道去哪里寻找当前项目需要的类。

但 BuildPath 可不止只有这个管理 类库的操作，他还有很多的东西，我们打开某个工程的 Buildpath设置 Build Path—>Configure Build Path，可以发现有如下的操作

一共有四个选项卡：下面简单介绍一下它们分别是干啥的

Source

指定源文件路径 和 输出文件路径。源文件路径就是我们的 .java 文件所在地址嘛，而输出文件路径对应的就是保存 编译后的 .class 文件啦。

比如你创建新项目的时候，就会让你指定好输出的路径 (普通的Java工程默认指定: bin 目录)

如上图，默认指定的路径为 bin 目录，但是我们在 Eclipse 中是看不到的(我也不知道为什么，好像是被隐藏了)，不过在文件系统中能够发现。比如我们添加了一个 java 文件，Eclipse 其实就会自动调用 javac 命令编译产生 .class 文件(可以自己测试一下，添加一个 java 文件，然后再对应 bin 文件中查看情况)。
我们来模拟 Eclipse 的编译操作：指定编译的 java 路径，同时也指定输出的 .class 文件的路径，其实就是模拟 Eclipse 指定的 src 和 bin 目录啦。

Library

Libaray 就是管理当前工程引用的 系统运行库（JRE）、第三方的功能扩展库、工作空间中的其他工程，甚至外部的类文件。

Add JARs：是添加当前项目目录中的的jar文件。
Add External JARs：是添加当前项目之外的外部目录中的jar文件。
Add Variable：是添加变量。
Add Library：是添加JRE系统的库。
Add Class Folder：添加当前项目中编译之后的.class文件。
Add External Class Folder：添加当前项目之外的.class文件。

HL瞎BB

其实我对于这个 Library 理解就是字面的意思：类库。它存放了所有编译和运行时期需要的 .class文件(你可以把 jar包理解为 .class 文件的集合)。

想想最开始所举的 Buildpath 的栗子，Eclipse 需要一片空间来存放所有当前工程之外的类库，而这片空间就是 Library 了。在我们想在 Eclipse 中引用一个 jar包的时候，就需要将其添加到 Eclipse 专门存储的地方 —— Library，这样在编译的时候 Eclipse 就知道：奥，你想引入这个类，我去我的类库给你找找吧。

至于上面的各种添加形式，简单就是导入类的形式不同咯，它们本质都是一样的，都是将工程之外的需要引用的类放到 Eclipse 专门存放地方，方便 Eclipse 进行管理。

有个疑问

在我们新建工程的时候，有个地方能够让我们选择 JRE

这其中可以选择配置 JRE

最上面说到，默认情况下，下面被勾选的 JRE 将会在新的项目创建的时候加入到 Buildpath 中。

然后我们在选择编辑 JRE 的时候可以当前 JRE 的 HOME 目录，以及其中包含的包。

所以我现在有个这样的问题，当我们使用 IDE 的时候，创建项目的同时我们就将 JRE 添加到了项目中，也就是说，Eclipse 帮我们在创建项目的时候就加入了 JRE 这个类库，所以我们才能够创建一个 JRE 类库中的类(比如 System 类)，并且使用 System.out.println("hello world~"); 输出语句输出 HelloWorld。

但是在我们脱离了 IDE 之后，通过一些编辑器编写，命令行的形式来运行 Java 代码。这种情况下谁来帮我们帮我们添加运行时候的类库呢？

我们先试试将项目中的 JRE 这个类库删除之后，看看会报出什么错误

可以发现我们删除 JRE 类库之后，各种报错，Object、String、System 都找不到了，这是正常的反应，因为我们 JRE 类库删除了，这些类都是存在于 JRE 类库中的。

但是我们的命令行呢，为什么我没有手动加入JRE包编译也不会报错呢？为什么还能够正常运行？是在那个阶段进行类库的加载呢？在哪配置了 JRE的默认加载？通过谁来加载外部 Jar包？

这个问题先立个 Flag，等后面有机会了解一下这方面的知识。

Order And Export

首先看这个标签页的名称能得出的信息是：它的用作是 修改顺序 和 导出设置。

先看看 Eclipse 官方给出的解释

在 Order And Eport 中你可以修改这些已经选择的库的Buildpath顺序。被选择的库被标记为导出之后，当某一项目依赖与这个导出的项目的时候，这些被标记为导出的类库就对此项目是可见的。Source 文件夹一定会被导出，且不可修改为不导出。

当我们选择勾选的时候，这个 classpathentry 就会多出一条属性：exported="true"。我们再结合着官网文档 和 Stack Overflow 的两个回答：

Exported entries are visible to projects that require the project.

A dependent project that expects some resource from the contributing project will not compile/run if the contributing project doesn’t export that resource.

我们可以得出以下结论
如果一个项目不选择导出它自己的资源，则其它依赖于此项目的项目都无法使用这个没有被导出的资源。

举个简单例子(其实就是上面 5票的答案)

ProjectA 存在类库 LibA(可以是自定义的类库、一个 Jar包、或是一个文件夹，只要是能够被勾选的都算)。现在 ProjectB 依赖于 ProjectA，并且想使用它的 LibA 来做事情，但在导出的时候，ProjectA 选择不导出 LibA，这时候 ProjectB 就无法编译/运行了。

原理上分析大概就是如此，但是实际我也不知道怎么进行测试，感觉上就只是跟 Eclipse 这个IDE 有关，就是 Eclipse 的一些项目管理的规定吧，我猜是这样哈，总之感觉跟 Java 没什么关系，所以这里就了解到这里了……
