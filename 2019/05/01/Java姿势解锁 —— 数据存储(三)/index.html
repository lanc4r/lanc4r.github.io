前言

剩下就还有方法区没有讲啦，这次聊聊方法区，那么 Java 数据存储这一块就先告一段落啦。

方法区

简介

方法区又称为静态区，它存放着程序运行时一直存在的数据，它同Java 的堆一样，是各个线程共享的内存区域。

我们来看看官方的解释：

The Java Virtual Machine has a method area that is shared among all Java Virtual Machine threads. The method area is analogous to the storage area for compiled code of a conventional language or analogous to the “text” segment in an operating system process. It stores per-class structures such as the run-time constant pool, field and method data, and the code for methods and constructors, including the special methods (§2.9) used in class and instance initialization and interface initialization.
The method area is created on virtual machine start-up. Although the method area is logically part of the heap, simple implementations may choose not to either garbage collect or compact it. This specification does not mandate the location of the method area or the policies used to manage compiled code. The method area may be of a fixed size or may be expanded as required by the computation and may be contracted if a larger method area becomes unnecessary. The memory for the method area does not need to be contiguous.
A Java Virtual Machine implementation may provide the programmer or the user control over the initial size of the method area, as well as, in the case of a varying-size method area, control over the maximum and minimum method area size.
The following exceptional condition is associated with the method area:
· If memory in the method area cannot be made available to satisfy an allocation request, the Java Virtual Machine throws an OutOfMemoryError.

大致说的是

JVM中的方法区被所有的 JVM 线程共享。方法区和编译传统语言的存储区类似，或者是同操作系统进程的文本部分相似。它存储了每个类的结构信息：运行时的常量池，域和方法数据，同时包括方法(包括被用在类、实例、接口初始化的特殊方法)和构造函数的代码。
方法区在 JVM 启动的时候就会被创建，尽管方法区在逻辑部分上是属于堆的，但是它的数据不受 GC 的控制，同时也不会被压缩。这个规范并不会决定方法区的位置或是管理编码的策略。方法区可能在初识化时候就有一个合适的长度，并且它也可以在不合适时候的随意变换自身的大小。方法区中的内存也并非需要是连续的。
一个JVM的实现可能提供给开发者或是用户能够控制方法区初始大小的能力，同时在改变方法区大小的情况下，控制方法区大小的临界值(最大、最小值)。
在特殊情况下，如果当前请求需要分配的方法区内存不能保证安全的话，就会抛出内存溢出错误。

注意，这里提到，方法区存储了每个类的结构信息，它们包括以下几个部分

  • 运行时的常量池
  • 域
  • 方法的数据
  • 方法和构造函数的代码

方法区存放的数据

上面说了方法区储存的类的结构信息主要包括四个部分，下面我们依次来聊聊~

方法区存储的运行时常量

Java 中所有 static final 修饰的成员变量都存储于方法区的运行时常量池中。这个就不多说，你看到 final 修饰的变量一定是存储在常量池中的就对了。

方法区存储的域

这里的域其实就是我们的类变量啦，也即是 static 修饰的值嘛。

我么之前讲堆栈的时候说过，类的成员变量依附于实例，所以它们存储在堆中。但是如果这个成员变量被 static 关键字修饰之后，他就变成了类变量，和类关联在一起，不依附于实例，所以不再存储在堆中。

方法区存储的方法数据

程序运行时会加载类编译生成的字节码，这个过程中静态变量（类变量）和静态方法及普通方法对应的字节码加载到方法区。
但是！！！方法区中没有实例变量，这是因为，类加载先于对应类对象的产生，而实例变量是和对象关联在一起的，没有对象就不存在实例变量，类加载时没有对象，所以方法区中没有实例变量
静态变量（类变量）和静态方法及普通方法在方法区（Method Area）存储方式是有区别的

上面的内容摘抄自某位博主的博文(在底部的 Ref 中能够找到)，总之你理解为无论是 静态方法 还是 非静态方法都会加载到方法区就完事了，至于最后博主说了 静态变量、静态方法、普通方法在方法区的存储是有区别的，但是到底有什么区别啊，你倒是说说啊….

方法区存储的类信息

以下是摘抄自之前博主的内容：

对每个加载的类型（类class、接口interface、枚举enum、注解annotation），JVM必须在方法区中存储以下类型信息：

  • 这个类型的完整有效名称（全名=包名.类名）

  • 这个类型直接父类的完整有效名称( java.lang.Object除外，其他类型若没有声明父类，默认父类是Object)

  • 这个类型的修饰符(public、abstract、final的某个子集)

  • 这个类型直接接口的一个有序列表

这些东西其实很难放到我自己列出的类结构的那几点中去，我感觉是我自己的翻译的问题，所以这里强行把它加入进来了，因为真的很重要。

针对博主说的这些信息说说自己的理解：

因为方法区存储的数据是运行时候会调用的，那么调用的时候需要某个类，你肯定是需要知道这个类的路径嘛，那么这个类的完整包名必然是需要存储的；再比如上面说的父类，你再创建一个对象的时候，肯定是先调用这个类的父类的构造方法，所以父类的信息也是必须要的。
