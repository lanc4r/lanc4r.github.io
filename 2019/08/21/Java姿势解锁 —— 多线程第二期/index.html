Lock 初识

Lock 概念

Lock 是一个 JDK 1.5 版本增加的一个接口，它和之前的 synchronized 关键字的作用都是一样的，解决线程同步问题(synchronized 参见 Java 姿势解锁 —— 多线程第一期)。Lock 在使用时需要显式地获取和释放锁，缺点就是缺少像 synchronized 那样隐式获取释放锁的便捷性，但是却拥有了锁获取与释放的可操作性，可中断的获取锁以及超时获取锁等多种 synchronized 关键字所不具备的同步特性。

接口和实现类

作为初识我们先了解常用的三个接口和一个实现类，更多的资料请参考下方 REF 提及的文章。

接口

  • void lock() 获取锁，调用该方法当前线程会获取锁，当获取锁后，才有资格执行后续代码，你可以理解为调用 lock() 方法就会停在 synchronized(obj){ ... } 方法块处，当获取到锁后就相当于进入到 synchronized 方法块中执行对应代码。
  • void unlock() 释放锁。这个理解为之前的 synchronized(obj){ ... } 方法块执行完后释放对象锁的操作，只不过之前是隐式的，现在需要我们显示调用
  • boolean tryLock() 尝试非阻塞获取锁，调用该方法后立即返回，如果能够获取锁则返回 true，否则返回 false。这个就是之前 synchronized 关键字所不具备的功能啦，这里有一个类似“跳过”的功能。
  • boolean tryLock(long time,TimeUnit unit) 超时获取锁，当前线程在以下3种情况返回：
    • 1.当前线程在超时时间内获取了锁
    • 2.当前线程在超时时间被中断
    • 3.当前线程超时时间结束，返回false

实现类

ReentrantLock 重入锁，顾名思义就是支持重新进入的锁，它表示该锁能够支持一个线程对资源的重复加锁，也就是说在调用lock()方法时，已经获取到锁的线程，能够再次调用lock()方法获取锁而不被阻塞，同时还支持获取锁的公平性和非公平性。这里的公平是在绝对时间上，先对锁进行获取的请求一定先被满足，那么这个锁是公平锁，反之，是不公平的。那么该如何使用呢？看三种情况。

1. 代替 synchronized 关键的用法(上面说过了)

2.防止重复执行代码

3.尝试等待执行的代码

条件对象的等待/通知机制

所谓的条件对象也就是配合前面分析的 Lock 锁对象，通过锁对象的条件对象来实现等待/通知机制。这个对应于 synchronized 就理解成对象锁啦，看看下面的 API 就明白了。

条件对象API

  • void await() —— 将该线程放到条件等待池中（对应wait()方法）
  • void signal() —— 从该条件的等待池中随机地选择一个线程，解除其阻塞状态（对应notify()方法）
  • void signalAll() —— 解除该条件等待池中所有线程的阻塞状态（对应notifyAll()方法）
