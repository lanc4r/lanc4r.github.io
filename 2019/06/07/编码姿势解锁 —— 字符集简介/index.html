编码姿势解锁 —— 字符集简介

发表于 2019-06-07 • 分类于 流下了没有技术的泪水 , 编码姿势解锁

前言

找到了一篇老博客，记录的时间是 2016年11月份，这里翻出来重新学习一下。

简单概念

这篇主题是字符集，我们先了解几个概念

字符

字符是抽象的最小文本单位。它没有固定的形状（可能是一个字形），而且没有值。“A”是一个字符，“€”（德国、法国和许多其他欧洲国家通用货币的标志）也是一个字符。“中”“国”这是两个汉字字符。字符仅仅代表一个符号，没有任何实际值的意义。

字符集

字符集是字符的集合。例如，汉字字符是中国人最先发明的字符，在中文、日文、韩文和越南文的书写中使用。这也说明了字符和字符集之间的关系，字符组成字符集（常见的有：Ascii、ISO8859-1，GB2312/GBK，Unicode）。

代码点

字符集中的每个字符都被分配到一个“代码点”。每个代码点都有一个特定的唯一数值，称为标值。该标量值通常用十六进制表示。

代码单元

在每种编码形式中，代码点被映射到一个或多个代码单元。“代码单元”是各个编码方式中的单个单元。代码单元的大小等效于特定编码方式的位数。

字符集和编码

在之前我经常分不清楚 UTF-8 和 Unicode 的关系，其实它们的关系就是 编码和字符集的关系。
Unicode 是一种包含全世界所有字符的字符集，又叫统一码、万国码，它为世界650种语言进行统一编码，你简单理解世界上所有的语言字符都可以在其中找到。
UTF 全称是：Unicode Transformation Format，翻译过来就是 Unicode 编码方式，那么 8 呢，其实表示的是 Unicode 的其中一种编码方式，它规定最小的是单位是 8 bit，也即是 1 byte，这就造就 UTF-8 是变长的属性。当然 Unicode 的编码方式
还包含 UTF-16、UTF-32，它们最小的单位分别是：16bit 和 32bit。

你可以这样理解，想想最熟悉的 ASCII 码表，其中一个字符对应着一个数字，比如小写字母 a 对应就是 97。其实 Unicode 和 ASCII 是一样的，也是一个字符对应着一个数字，只是 Unicode 这个表大一些罢了。

至于编码，其实就是对这个字符对应的数字编码，比如一个汉字 “瞬”，它对应着 Unicode 的值是 ： 30636，这时候用 UTF-16 编码为 0x77AC，但如果通过 UTF-8 来编码的话结果就不是 0x77AC 而是 0xE79EAC。不管是 0x77AC 还是 0xE79EAC，它们最后解码后一定会得到值：30636。

所以编码其实就是把字符集中字符对应的数值以特殊的形式保存起来，这样做有一些好处，比如节省存储空间，增加容错性，这个待会说~

计算机中的存储

我们知道计算机中是二进制的世界，那么它是怎么存储我们的字符的呢？当然就是存储字符对应的数值啦，其实也就是我们的代码点，如果字符被编码，它只需要存储编码后的值就行啦。比如“瞬”在计算机中存储为 0x77AC，或是编码后的样子 0xE79EAC。

思考

这里还有一个问题，我们只是存储了其代码点，但是如何从代码点映射到相应的字符呢？比如 Unicode 字符集，这个字符集到底存储在哪里呢？

常见的字符集和编码

Unicode

这次我们只讲最常见的字符集 Unicode，前面说过它是包含全世界所有字符的集合。在最开始的时候它只用 2个字节来编码字符，也即是 2的16次方 65535 个字符。但这样一个纷繁复杂的世界，只用 2个字节显然不能表示所有的字符，于是为了包罗万象，Unicode 的制定者 The Unicode Consortium 扩展了 Unicode，使用 4个字节来表示(大概有 20亿+)，比如我们常用的 emoji 表情，它也被归纳进 Unicode 字符集中，这个大小应该能撑到人类文明灭亡那天吧….

Unicode 编码

UTF-32

UTF-32 编码最小单位是 32 bit，也即是 4 byte。那么它是怎么存储字符的呢，举个栗子：字符 a，它的对应数值是 97，也即是 0x61，它占据的大小是 1个字节，如果我们使用 UTF-32 来存储字符a，却需要 4个字节来存储！因为它的最小的单位是 4个字节嘛，这样就会白白浪费掉 3个字节的数据。

UTF-16

同理 UTF-32，它的最小单位是 16bit，2byte。它的存储方式就是2字节个一存的，所以它相对于 UTF-32 来说是变长的。比如某个字符所占大小是 3个字节，如果使用 UTF-16 编码来存的话，最终的大小就是 4个字节，因为它总是 2字节的递增的嘛。

UTF-8

UTF-8 最小单位是 8bit，1 byte，它通过单个字节的递增来达到变长的特性。比如我们在最开始时候说的字符 a，如果使用 UTF-8 来存储它的大小就是 1byte。下面我们来看一下不同大小的数值在 UTF-8 中是怎么存储的。

Unicode 第一版

Unicode第二版

因为第一版的 Unicode 只能存储2个字节的数据，到第二版后，UTF-8 最大字节数也从 3个字节变到 6个字节

从上面的情况我们可以看出，因为 UTF-8 为了实现变长的功能，它牺牲了一些位置做了标志位，待会再说为什么这里需要这种标志位。所以这里导致了一个问题，比如一个大小为 0x8F的字符，它本来只会占据一个字节，但是如果使用 UTF-8 来存储的话就会占据了两个字节。

还有如果我们想存储的数据大部分都是汉字的话，此时使用 UTF-16 编码就比 UTF-8 会节省空间，因为汉字的编码大多在 0x3FFF 左右(在 U-0800 - U-FFFF 范围内)，这时使用 UTF-8 来存储的话就是三个字节，但是使用 UTF-16 编码就只会使用两个字节啦。所以 UTF-8 也并不总是最好的，如果你是想要节省空间，最好是根据所存储的数据来选择编码方式；但如果不是那么在乎存储空间，还是推荐使用 UTF-8 的编码方式，因为普遍啊，大家都用，到你这儿也不用再去转码。

UTF-8 的标志位

我们知道 UTF-8 是变长的编码格式，也就是一串字符，可能第一个占据 1个字节，第二个占据 2个字节，第三个占据 3个字节。那么它怎么知道当前的这个字符应该占据几个字节呢，这就是标志位的作用啦，比如当它第一次开始读的时候，发现第一个字节数据是 110xxxxx 的形式，它就知道了：奥，当前这字符需要两个字节，于是再读后面一个字节，将当前两个字节进行解码得到字符。
如果是 1110xxxx 的形式，它就再读后面两个字节，将三个字节数据进行解码得到字符。

至于后面为什么是 10xxxxxx 的形式我就是猜的了，你看看决定读取字节个数的标志位，都是 11开头的，比如 110、1110，这里使用 10 刚好可以和标志位区分开来：如果要是不写的话，那么这个字节的大小可能正好是 110xxxxx，这时候又会把其当做第一个字符来读取了，这样就不对啦。

乱码的形成和容错

乱码的形成主要原因就是编码方式不一致嘛，比如你一个 UTF-8 编码的字符串：(0x2F)(0xE7A3A7)(0xD8A9)，这时你使用 UTF-16 编码来解码的话，就会变成 (0x2FE7)(0xA3A7)(0xD8A9) 这时候当然解出来的字符和之前 UTF-8 想表达的字符就不同了啦。并且 UTF-8 编码中，每个字节都会有标志位的存在，这些在解码的时候是需要被舍弃掉的(具体怎么做看后面编解码部分)，如果另一种编码方式将标志位也当做字符编码的一部分，这样得到的数据可定会有所不同。

接下来说说容错的事情，这里就以 Unicode 的第一个版本来说明吧，此时 UTF-16 编码是定量读取两个字节，两个字节足够表示第一版所有的字符，所以此时，UTF-16 编码是没有标志位可言的。

比如一个字符串为：(0x2FE7)(0xA3A7)(0xD8A9)(0xA78D)，这时候在传输的时候第二个字符的一个字节(0xA3)丢失了，这时候 UTF-16 解码就变成： (0x2FE7)(0xA7D8)(0xA9A7)(0x8D) 红色的部分开始，之后的字符串将会全部乱掉。

再来说说 UTF-8，我们知道 UTF-8 读取字符串的时候是通过标志位来判断的，也就是说就算你在传输的过程中丢了一个字节，唯一会影响的就是涉及到那个字节的字符而已，对于于后的字符，同样还是会根据标志位来判断读取，后面的字符并不会受到影响。

由上可以得出结论：UTF-8 相较于 UTF-16 的容错性更好，你可以简单的理解为拥有标志位检测的编码容错性都不错。

前面我们说过，UTF-8 就是把 Unicode 中的字符对应的数字进行存储，这个存储的过程就称为编码，同理把 UTF-8 存储的数字取出来就称为解码咯。

小彩蛋

USC、USC-2、USC-4

USC 全程是 Universal Multiple-Octet Coded Character Set，你可以理解也是一套字符集，它是由 ISO 组织开展的一个项目的产物。它和 Unicode 一样也是一个字符集，到 Unicode2.0 时候，USC 和 Unicode 编码基本一致。

USC-2 和 USC-4 是 USC 的编码方式，前者使用 2个字节编码，后者使用 4个字节。

具体请参考 unicode和ucs的区别

字节序

字节序是指在编解码的时候通过不同的顺序来进行，什么意思呢，还是来看看 “瞬” 字，我们通过不同的编码形式来保存。

由于历史原因，windows 默认把 UTF-16 little Endian 叫做了 Unicode 有兴趣的可以去了解一下 知乎。

我们来看看其十六进制数据，发现它们的顺序改变了，并且要注意文件头，两种编码方式的文件头是有区别的。在进行网络传输的时候，通过字节序来决定把第一个接受到的字节放到高位还是低位。

比如一个字符编码是 0xAC9D，它传输过程中第一个接收的字节是 0xAC，那么把这个字节放到高位得到结果就是 0xAC9D，但如果把第一个字节放到低位此时得到结果就是 0x9DAC。

LE 是 Little Endina 的缩写表示低位开始，BE 是 Big Endina 的缩写，表示高位开始

UTF-8 没有字节序

我猜是因为 UTF-8 存在标志位啊，字节序是表示你先接受到这个字符是作为高位存放还是地位存放，这只是在所有位都表示字符的时候才会起作用，如果有些是标志位，这时候我们改变一下字符的顺序，再次解析的时候就会出问题啦！
