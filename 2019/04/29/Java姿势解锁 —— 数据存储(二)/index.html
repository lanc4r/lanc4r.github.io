Java姿势解锁 —— 数据存储(二)

发表于 2019-04-29 • 分类于 流下了没有技术的泪水 , Java姿势解锁

前言

在上一篇介绍存储区域的文章中，我们介绍了常量池，这次聊聊堆栈的相关内容~

堆内存 和 栈内存

简介

栈内存：位于 RAM 中，就是我们常听到的栈。有一个栈顶指针，通过指针移动进行压栈和出栈，从而分配和释放内存。这种方式的存储速度仅次于寄存器。Java中常用于存放 对象引用 和 基本数据类型，而不用于存储对象。

在栈内存中，要求存放入其中的数据都是能够确定大小的，并且生命周期也必须是能够确定的。

堆内存：位于 RAM 中的一种通用内存池。其中存放的数据由 JVM 自动进行管理。堆相对于栈的好处来说：编译器不需要知道存储的数据在堆里存活多长。当需要一个对象时，使用 new 写一行代码，当执行这行代码时，会自动在堆里进行存储分配。同时，因为以上原因，用堆进行数据的存储分配和清理，需要花费更多的时间。

在堆内存中，其数据的存放大小可以被 JVM 动态的分配，所以无需事先告诉编译器数据的大小、生命周期等相关信息。

注意

堆与栈都是位于内存上的，并且它由 Java 自动化管理，不能人为去直接设置。

栈内存和堆内存的比较

存储的数据类型的比较

栈内存存放局部变量(基本类型和引用对象)，而堆内存中存放着对象(实体)

先说说栈

因为我们知道栈中需要存储的是能够确定大小和生命周期的东西，那我们就来看看到底什么东西是能够确定大小和生命周期的，注意这个确定时期是指在编译的时候，并不是在运行的时候，因为运行的时候肯定知道大小了嘛….

首先是 8 种基本类型，它们实际上都是存储了一些占不同字节的二进制数据，所以只要知道了变量的类型就能知道这个变量占据多少个字节。
然后我们来看看引用对象，引用对象存储的是对象在内存中的地址，其实这个内存地址也是一串二进制的数据值(就像这样: java.util.Random@4e25154f)，所以它的大小也是固定的。

在程序运行中，只有我们的局部变量是能够确定时间的，比如方法内定义的变量、方法的参数，它们随方法进栈执行，然后到方法执行完毕之后出栈，这时候局部变量的生命周期结束。

所以为了满足栈的要求，也只有局部变量中的基本类型和引用变量存储在栈中。后面会有一个例子来演示的。

再说说堆

我们使用关键字 new 调用构造函数创建的类的对象数据都存在于堆中(只不过它的引用存在了栈中)。因为对象的数据在编译期是不会在内存中进行创建和存储的，只有在程序运行的期间才会根据需要进行动态的创建和存储。

并且其释放资源的途径就是由 Java 的垃圾回收器来做的，我们根本不知道它什么时候会被回收，所以其生命周期也是不确定的。

因为在程序运行之前，我们不能确定对象的大小、生命周期等内容，所以对象数据满足了堆的存储要求，它们存储于堆中。

存储速度的比较

栈内存的分配与清理速度都大于对内存，并且栈内存的存储速度仅次于寄存器

先说说分配速度

这个其实很好理解，一种数据给定了大小，一种数据没有给定大小，这意味前者你可以提前就做好准备，比如 5kb 大小，你发现诶，这儿正好有个 5k 大小的空间空着，那就放这吧。而后者是没有给定大小的，这意味着你需要在运行时候，得到具体大小之后再慢慢去找位置存放，这样肯定是需要花费更多的时间的。

再说说清理速度

这个就更好说了，一个是知道生命周期的，只要一出栈，直接清理掉，都不带犹豫的；而另一种数据是由垃圾回收器来释放，可能由于一些优化机制，垃圾回收器的执行效率可能会受到影响，导致清理的速度减弱(来自HL的瞎BB)。并且就数据量而言，栈中确定大小的数据也肯定比堆中不确定大小的数据清理的快(大部分情况都是堆中的数据量大于栈中的数据量)。

灵活性比较

由于栈和堆的存储机制的差异，导致堆内存的灵活性大于栈内存。

这个就不用多说了，毕竟一个是固定好的，给了你一堆条条框框的限制(栈)，还有一个充满着自由♂的气息(堆)。freedom is awesome~
