如何实例化大量参数的类

这里主要介绍三种方式

1. 使用 重叠构造器 来创建此类

在这种模式下，你需要提供 一个只有必要参数的构造器，第二个构造器有一个可选参数、第三个构造器有两个可选参数….

举个栗子来说明这种情况的缺点，假如我们只想创建一个不含有 fat 属性的实例，并且需要确保其他属性的值都是存在的，这个时候我们就需要这样调用：

NutritionFacts nutrition = new NutritionFacts(111, 111, 111, 0, 111, 111);

咋一看其实还没有多复杂对不对，但是当我们有 20个属性，有其中 5个属性是不需要设置的，这个时候如果继续使用这种方式来构造，一下子传递 20 个值，根本很让人崩溃好吗，读者需要一个一个的对应着接口上的名称配合注解，才能知道某个参数的作用是什么。如果又换另外 5个属性不需要设置值…

还有一个情况，如果类型相同的参数，不小心颠倒了顺序，这个时候编译器不会出错，只有在程序运行的情况才能发现，Debug 也会浪费大量的时间….

简单的说：重叠构造器模式是可以用的，但是当有许多参数的时候，客户端代码会很难编写，并且仍然较难以阅读。

2. 使用 JavaBean 模式

JavaBeans 模式实际上是：调用一个无参构造器来创建对象，然后调用 setter 方法来设置每个必要的参数，以及每个相关的可选参数，依然通过上述的栗子简单了解一下：

这种模式弥补了重叠构造器模式的不足之处。简言之，就是创建实例变得简单了，产生的代码也易读。

最主要的原因就是 构造过程被分到了几个调用中去了，在构造过程中 JavaBeans 可能处于不一致的状态。试图使用处于不一致状态的对象，将会导致失败。

此外还有一个缺点： JavaBeans 模式阻止了把类做成不可变的可能，这就需要程序员付出额外的努力来确保它的线程安全。

使用 Builder 模式

Builder 模式不会直接生成想要的对象，而是让客户端利用所有必要的参数调用构造器(或者静态工厂)，得到一个 builder对象。简单总结一下其顺序：

  1. 首先客户端调用一个需要所有必要参数的 构造器(或者是 静态工厂)，返回一个 builder 对象
  2. 然后 客户端通过 builder 调用类似的 setter 方法，来自定义可选参数
  3. 最后客户端调用无参的 builder 方法来生成 不可变对象

约束性检查

使用 Builder 模式依然可以传入参数的约束性检查，一般有三个地方可以进行检查：

  1. 调用 Builder 类 内部的 build() 方法的时候，我们会将 Builder 类的对象传递到 NutritionFacts 类中去，在传递之前我们可以对 Builder 对象自身进行检测
  2. 在 执行 NutritionFacts(Builder builder) 方法的时候，我们可以对 传入的 Builder 对象的值进行检测，如果有什么不合法的参数可以直接在这里的过程检测出来
  3. 在 Builder setter 方法(或者是 构造器方法) 中，每次传入参数的时候都可以进行检测，这是最根本的检测参数异常的地方。

Builder 模式的缺点

Builder 模式当然也是存在不足之处。为了创建对象，我们必须先创建它的构建器。虽然创建构建器的开销在实践中可能没有那么明显，但是在某些十分注重性能的情况下，这部分开销可能就没有办法被忽略了。

此外，Builder 模式还比 重叠构建器模式 更加冗长，因此它只有在很多参数的时候才会使用。一般在 4+ 个参数时候会选择使用构建器的形式。并且还要记住，将来你有很大的可能会扩充参数的个数，如果一开始选择的是 构造器 或者是 静态工厂方法，得到类需要增加参数的时候才添加构建器，这个改动就会很蛋疼了，你需要将之前所有创建该类实例的地方全部修改，我的妈，太可怕了，几乎是不可能的，所以在前期是否该选择使用构建器需要仔细斟酌！
